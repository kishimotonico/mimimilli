# ADR-0006: モバイルUIは「再生とマークだけ」の共有URL・シェル分岐で作る

- ステータス: 承認（未実装）
- 日付: 2026-07-11
- 関連: [../design-system.md](../design-system.md)、backlog TASK-30 とモバイル関連ドラフト群

## 文脈

UIは完全にデスクトップ前提の3ペイン構成（LeftNav・リスト・インスペクタ＋プレイヤー）で、`@media` クエリはほぼ存在しない。スマホからも使いたいが、PCが母艦であることは変わらない。ライブラリ管理（タグ編集・スマートフォルダー編集・スキャン等）はPCで行う前提のため、スマホには管理機能を持ち込まず、「ストリーミング再生するプレイヤー」と「作品を探す道具」に徹する。

## 決定

### スコープ

スマホUIの初期スコープは以下のみ。管理系UI（タグ編集・スマートフォルダー編集・スキャン・設定変更）は持ち込まない。

- 再生: ミニプレイヤー（画面下固定）＋タップで全画面プレイヤー展開。MediaSession API でロック画面・イヤホン操作に対応し、画面オフでも再生継続
- ブラウズ: カバー主体のグリッドと、PCで作成済みのスマートフォルダーを入口とするポータル。検索はタグタップ絞り込み中心
- マーキング: 「お気に入り」「あとで整理」の2つのトグルのみ（後述）

続きから再生（再生位置のサーバー保存）は初期スコープ外。次フェーズの最有力候補として backlog にドラフトを置く。

### URL・ルーティング

URLは分けない（`/m` のような別ツリーは作らない）。ルーティングは既存の `navigationUrl.ts` ＋ `useNavigationHistory` をそのまま共有し、分岐はビュー層のみで行う。

### シェル分岐方式

- ブレークポイントは 768px の単一分岐（モバイル/デスクトップの二値）。値は `tokens.css` にトークンとして定義し、JS側は共有の `useMediaQuery` 系フック経由で参照する
- 狭幅時は `AppShell` の代わりに `MobileShell` を新設して `App.tsx` で出し分ける。データ層・features の model・スロットに渡す中身は共有し、モバイルで意味を持たないスロット（LeftNav・AddressBar 等）はボトムタブ＋簡易ヘッダーに置き換える
- デスクトップの複雑なコンポーネントに `md:` ユーティリティを散らして畳む方式は採らない。操作系が根本的に異なるもの（プレイヤー等）はモバイル用ビューを別コンポーネントとして持つ

### ナビゲーション形態

ボトムタブ（ライブラリ / スマートフォルダー / お気に入り の3タブ想定）。ミニプレイヤーはタブバーの直上に重ねる。

### マーキングのデータモデル

作品に独立フィールドは追加せず、既存のタグ機構で表現する（フラットタグ「お気に入り」「あとで整理」）。

- ユーザー目線で仕様が単純になり（マーク＝タグ）、スマートフォルダーのタグ条件でお気に入りビューや整理キューがそのまま作れる。ポータル方式と整合する
- スマホUIではタグ編集UIは出さず、この2タグに限定したトグルボタンとして見せる
- 既存のタグ更新APIを流用でき、shared の契約変更が不要

### 配布形態

PWA（manifest 追加・ホーム画面追加・スタンドアロン表示）。オフラインキャッシュ、外出先アクセス（Tailscale 等）、帯域向けトランスコードは初期スコープ外。

## 帰結

- 既存タスクとの整合: TASK-6（900px狭幅）はデスクトップレイアウト内の調整として本ADRとは独立に扱う（768px以下はモバイルに切り替わるため、900px問題はデスクトップ側の話のまま）。TASK-14（グリッド表示）はモバイルブラウズの土台になるため先行が望ましい。TASK-25（インスペクタ・オーバーレイ）はデスクトップ専用で、モバイルには持ち込まない
- マークをタグにしたため、「お気に入り」「あとで整理」タグの改名・削除は通常のタグ操作で可能（保護はしない）。運用で困る事態が起きたら予約タグ化を検討する
- `PlayerDock` の bar/popup モードにモバイル用ミニプレイヤーを混ぜるか、別コンポーネントにするかは実装時の判断に委ねる（本ADRはシェル分岐の原則のみ規定する）
- 実装タスクは backlog のモバイル関連ドラフト（シェル・プレイヤー・ブラウズ・マーキング・PWA・ビジュアルテスト。ドラフトIDは変動するため名前で参照）に分割済み
