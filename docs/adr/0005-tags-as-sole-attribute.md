# ADR-0005: タグを唯一の属性機構とし、特別扱いは設定データで表現する

- ステータス: 承認（実装済み）
- 日付: 2026-07-11
- 関連: [ADR-0004](0004-core-functions-over-sql.md)、[../requirements-v4.md](../requirements-v4.md) §4

## 文脈

本アプリでは CV・サークルといったほぼ必須の属性も、`お気に入りA` や `year/2025` のようなユーザー独自の属性も、すべて同じ「タグ」で管理・検索する（requirements-v4.md §4.2）。CV が複数いる作品も組み合わせで検索でき、属性の表現をシステムでなくユーザーが主体的に決められることが狙い。

データモデルはこの思想どおりタグ一元（`WorkSummary.tags: string[]`、`.meta.json` の `tags`）になっている一方、実装には思想からズレた特別扱いが散在していた。

- 分類軸が `circle / cv / series / cat / tag / year` のハードコード enum（`shared/src/library.ts` の `facetAxisSchema`）で、ユーザー定義 prefix は軸になれない
- 編集 UI が構造化タグ（`prefix/値` 形式）を一律編集不可にしており、DLsite 自動取得の誤りをユーザーが直せない
- タグの表示ラベル・色が UI コンポーネントにハードコード
- `genre/` タグは DLsite 連携で付与されるのに、どの軸にも表示されずブラウズ経路がない

「どこまでが便宜的な特別扱いとして許されるか」の線引きがドキュメントになく、実装が思想からズレても検知できない状態だったため、本 ADR で決定として固定する。

## 決定

### 1. タグは作品の唯一の属性機構

CV・サークルを含め、作品の属性をデータモデルの専用カラム・専用エンティティにしない。外部データの中間表現（例: `shared/src/dlsite.ts` の `circle` / `cvs` / `genreTags`）は例外とするが、境界（apply 時）で必ずタグに変換して合流させる。

### 2. タグは素の文字列であり、構造は慣習

`prefix/値`（Annotated タグ）は運用上の慣習であって型ではない。コアの検索・集計関数はタグ文字列を区別せず、prefix の解釈（`parseTag`）は表示・軸集計など必要な箇所でのみ行う。

### 3. 特別扱いはすべて「prefix 定義」という設定データで表現する

軸レールへの表示・削除時の保護・表示ラベル・色・表示優先度は、ユーザーが編集できる設定エンティティ（prefix 定義）のデータとして表現する。コードに特定 prefix の分岐を書かない。

システムの役割は初期値の投入のみ: 標準で用意する prefix は `cv` と `サークル` の2つ（DLsite 連携導入後は `genre` も初期投入候補）。投入後の変更・削除・追加はユーザーの自由。

分類軸はこの設定から動的に導出する（軸レール = 非タグ由来のビュー＋軸表示 ON の prefix＋スマートフォルダー）。`facetAxisSchema` の固定 enum は廃止する。

### 4. 保護はソフトガード

保護対象タグ（保護フラグの立った prefix に属するタグ）の削除・編集時は確認を挟むが、編集不能にはしない。保護の粒度は prefix 単位から始め、タグ個別の保護は必要が生じてから検討する。「構造化タグだから編集不可」という現行実装の区別は廃止する。

### 5. タグの同一性・正規化ルール

- prefix の比較は大文字小文字を無視する（`cv/x` と `CV/x` は同一 prefix）。保存時は prefix を正規形（小文字）へ寄せる
- 値は保存時に前後の空白を trim する。値の比較は大文字小文字を区別する
- 同義 prefix（`カテゴリ` と `category` など）のエイリアス解決は行わない。表記ゆれの統合はタグ横断操作（リネーム・マージ）で対応する

### 6. 絞り込みの一致セマンティクス

- 軸ドリル・タグ AND/OR 絞り込み・スマートフォルダーの `タグ ∋`: タグ全体の完全一致
- グローバル検索ボックス: 部分一致

現行の `filterByTags`（部分一致）はこの決定に合わせて完全一致へ改める。

### 7. 帰結として必要になる機能

- タグ横断操作（全作品を対象としたリネーム・マージ）。表記ゆれ統合の受け皿として将来フェーズで提供する
- ユーザーが削除したタグを外部連携（DLsite 再取得・自動適用）が復活させない仕組み（DRAFT-9 の連携状態モデル側で検討）

## 帰結

- 「CV もサークルもタグの1つ」という思想がコード上も真になり、ユーザー定義 prefix（例: `気分/`）が標準 prefix と同格に軸・保護・色を持てる
- `genre/` タグの受け皿問題は「初期設定で genre を軸表示にするか」という設定初期値の問題に還元される
- `facetAxisSchema` の enum 廃止は API 契約・URL 同期（`navigationUrl.ts`）・軸レール UI に波及する。既存6軸相当を初期設定として投入すれば利用上の互換は保てる。なお `year`（追加日）はタグ由来でない組み込み軸として分類軸グループに残し、ドリル絞り込みを `addedAt` の年照合に統一した（従来は集計が `addedAt` 由来・ドリルがタグ照合という不整合があり、常に0件になっていた）
- prefix 定義の保存場所（DB か、ポータビリティを重視してルートフォルダー直下の設定ファイルか）は要件決めタスクで確定する
- 全タグが編集可能になるため、誤操作に対する安全性は保護フラグ（確認ダイアログ）に依存する
