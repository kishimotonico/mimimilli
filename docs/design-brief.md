# mimikago デザインブリーフ

このドキュメントは、mimikago の UI/UX をゼロベースでデザインしてもらうための依頼書です。レイアウト・配色・タイポグラフィ・コンポーネントの粒度・ナビゲーション構造といった UI 上のあらゆる判断は、すべてあなた（デザイナー）に委ねます。本書は「このアプリが何で、誰が、何のために使い、何を譲れないか」だけを伝えます。

## 1. 期待する成果物

3 つの粒度のアウトプットを期待します。1 → 3 の順に具体化されていくイメージです。

1. **デザイン方針 / ムードボード**
   このアプリにふさわしい世界観を、言葉と参考ビジュアル（または抽象的なスタイル指定）で示してください。色のトーン、写真や質感の方向性、想起させたい体験（「夜の図書館」「個人のレコード棚」など）。
2. **デザインシステム**
   カラーパレット、タイポグラフィスケール、余白とグリッドの基準、よく使うコンポーネント（ボタン、入力、タグ、カード、トラック行、シークバー、モーダル、トースト等）の仕様。状態（hover / active / disabled / error）も含めて。
3. **画面ごとのデザインモック（HTML + CSS）**
   主要画面を一通り体験できる静的モック。フレームワークは任意、素の HTML/CSS でも Tailwind 等でも構いません。画面の取捨選択は委ねます（3 章のユースケースが一通り体験できることが目安）。

## 2. このアプリは何か

ローカルに保存した音声作品（DLsite や FANZA からダウンロードした ASMR / ボイス作品）を、自分一人で管理・再生するためのローカル Web アプリです。「再生するだけのプレイヤー」ではなく、**個人のコレクションを愛でながら、今夜聴くものを選んで、長時間没入する**ための場所です。

- 利用者は本人ひとり。マルチユーザーや共有・公開の概念はありません
- 主な利用シーンは **夜間、ヘッドホンで長時間**。画面はしばらく見続けるし、目に優しいことが大事
- 作品数は **数百〜数千規模**。1 作品あたりトラック数本〜数十本
- 主な利用環境は PC のデスクトップブラウザ。タブレット幅でも破綻しなければ嬉しい（必須ではない）

「再生するだけ」ではない点も重要です。作品フォルダーには音声以外のファイル（カバー以外のイラスト、特典 PDF、台本テキスト、紹介動画など）が同梱されることが多く、**作品フォルダー全体を快適に閲覧できるビューワー**であることも求められます。

## 3. ユーザーは何をしたいか（ユースケース）

優先度の高い順。

### 3.1 コレクションを眺めて、今夜聴くものを選ぶ

カバー画像を主役に、大量の作品を視覚的にブラウズする。「眺める楽しさ」が大事。タイトル・タグでの絞り込み、並び替え（追加日 / タイトル / 合計再生時間）が必要。表示密度はユーザーが調整したい（カバーを大きくゆったり眺めるモード ⇔ 多くの作品を一度に見るモード）。

### 3.2 作品の中身を確認して、特定のトラックから再生する

選んだ作品のメタ情報（タグ、URL、トラック構成、合計再生時間）をすぐ確認できる。任意のトラックを選んで、そこから再生を始める。「ちょっと中身を見たい」と「腰を据えて全部見たい」の 2 段階の動線がある（軽いプレビュー → 詳細）。

### 3.3 再生中、長時間（数十分〜数時間）快適に聴き続ける

一時停止 / シーク / 前後トラック移動 / 音量 / ±10 秒ジャンプ / 1 トラックのループ再生。再生中は **画面のどこにいても「今これを再生している」が常に把握できる**こと。再生中に他の作品を眺めたり検索したりすることもあり、その間も再生は途切れない。

ASMR の特性上、シーク操作の精度が体験を左右します。シークバーは十分な幅と当たり判定を確保したい。

### 3.4 作品フォルダーの中身を物理ファイル単位で覗く

トラックリストとは別軸で、作品フォルダーの中身そのものをブラウズしたい。

- フォルダーの階層構造をたどる
- 画像をプレビューする（特典イラストが複数枚あることも多い）
- 動画・テキスト・PDF の中身を確認する
- 音声ファイルもこの場から再生できる（トラック未登録のおまけ音声など）

### 3.5 作品を整理する

タグを追加・削除する。作品に紐づく URL を外部ブラウザで開く（販売ページや作者の SNS に飛ぶ）。

### 3.6 新しく追加したフォルダーを取り込む

ルートフォルダーを再スキャンする。新規に検出された作品候補を 1 件ずつ確認しながら登録する（タイトルを直す・スキップする）。

### 3.7 設定・初回セットアップ

ルートフォルダーの指定／変更、手動でフルスキャンを実行、前回スキャン日時の確認。初回起動時はルートフォルダー指定〜初回スキャンへ案内する。ここは凝らずに、迷わず通り抜けられれば十分。

## 4. 扱うデータ（情報構造）

### 作品 (Work)

- タイトル、カバー画像、追加日、物理パス
- **複数の URL**（ラベル付き。販売ページ・公式サイト・SNS など。通常 1〜2 件、多くて 5 件程度）
- **多数のタグ**（後述。数十個付くこともある）
- **1 〜 複数のプレイリスト**（SE 有無やフォーマット違いなどのバリアント。通常は 1 つだけ）
- **作品フォルダーに同梱される物理ファイル群**（音声・画像・動画・PDF・テキスト等）
- エラー状態（メタファイル不正、参照先音声ファイルの欠損、非対応フォーマット など）
- 再生位置・ブックマーク・最終再生日時（将来扱う情報）

### トラック (Track)

タイトル、再生時間、参照ファイルパス。単一ファイル形式の場合は再生区間（start/end 秒数）を持つ。

### タグ (Tag)

タグは 2 種類が混在し、**両者の違いが UI 上で一目で分かること**が重要です。

- **フラットタグ:** 単純な文字列。例: `バイノーラル`、`耳かき`、`R-15`、`触手`
- **Annotated タグ:** `カテゴリ/値` 形式。例: `cv/ひなたゆか`、`サークル/ほげふが`、`シリーズ/○○物語`、`genre/耳かき`

Annotated タグは「どのカテゴリに属する値か」が瞬時に判別できる必要があります。1 作品に 20〜30 個以上のタグが付くこともあるので、密度をどう扱うかが見せ場です。

## 5. 機能要件のハイライト（フェーズ1）

デザインの判断に効くものだけ抜粋。網羅的な機能リストは別途 `requirements-v4.md` を参照してください。

- **ライブラリ一覧:** グリッド表示。カバーサイズを 4 段階で可変（ゆったり眺める ⇔ 一望する）。エラーのある作品は一覧上で気付けること
- **検索・絞り込み:** タイトル / タグの部分一致。現在適用中の条件と結果件数が常に見えていること。並び替えは追加日 / タイトル / 再生時間
- **作品の軽いプレビュー:** 一覧から軽く中身を確認する動線（タグ閲覧・編集、トラック一覧、URL を開く、すぐ再生開始）
- **作品の詳細ビュー:** プレビューより広い面積でじっくり眺める。トラックリストと **物理ファイルブラウザ**（フォルダーツリー＋画像プレビュー）の 2 軸
- **プレイヤー:** 再生中は画面のどこにいても見える常駐 UI。クリックで全画面プレイヤーに展開し、カバー画像を大きく楽しめる。コントロールは 再生/停止・シーク・±10 秒・前後トラック・音量・ループトグル
- **スキャンと新規作品の取り込み:** 設定からフルスキャン起動。新規候補は 1 件ずつ確認するダイアログで登録 / スキップ
- **初回セットアップ:** ルートフォルダー指定 → 初回スキャン → ライブラリ一覧

## 6. 制約・前提

- **デバイス:** デスクトップブラウザ優先。タブレット幅でも破綻しなければ嬉しい
- **想定シーン:** 夜間、ヘッドホンで、長時間。**部屋の照明を落とした状態で眺めることが多い**
- **再生の常駐:** 一度再生を始めたら、別の作品を見たり画面を切り替えても再生は途切れない。「今何を再生しているか」が常に把握できる必要がある
- **キーボード操作:** スペースキーで再生/一時停止のトグルができると嬉しい
- **エラー作品の存在:** メタファイル不正やファイル欠損を持つ作品が一定数混ざる。一覧上で気付けること、詳細画面で具体的なエラー内容を確認できること
- **空状態:** 作品 0 件 / 検索ヒット 0 件 / スキャン未実施 などの状態があり、それぞれ「次に何をすればいいか」をユーザーに伝えたい
- **UI 文言:** 日本語

## 7. デザインで大事にしてほしいこと（ヒント）

- **目に優しいこと。** 夜間長時間の利用が前提。輝度差・コントラストの取り方に配慮を。ダークなトーンが自然に合うはずですが、ダーク前提とまでは縛りません
- **コレクションを「眺める」喜び。** カバー画像が主役。グリッドの粒立ち、余白、ホバー時の反応、ロード時の振る舞いに気を配ってほしい
- **Annotated タグの表現。** `cv/ひなたゆか` のような構造を、ただの文字列ではなく構造として見せる表現を探ってほしい。区切り文字を見せるか／カテゴリラベルを別要素にするか／色やシェイプで区別するか
- **常駐プレイヤーの存在感。** 邪魔にならず、しかし「今これが再生されている」が常に伝わる。一覧と全画面プレイヤーの間の中間表示として何が最適か
- **シーク操作の精度。** ASMR ではシーク位置の微調整が体験を左右する。当たり判定・視覚フィードバック・つまみの大きさ
- **多すぎる情報の整理。** 1 作品にタグが数十付き、ファイルも数十あり、URL も複数ある。情報密度をどう捌くか
- **エラー作品の扱い。** 一覧上で気付けることと、しかし全体の眺めを損なわないこと、この両立

## 8. デザイナーに委ねること / 委ねないこと

### 委ねること（自由に決めてほしい）

- 配色 / タイポグラフィ / 余白 / グリッドの基準
- 画面構成・ナビゲーション構造・遷移の作法
- コンポーネントの分割粒度、命名、状態表現
- 一覧の表示密度を切り替える UI、検索条件の見せ方、タグの表現方法
- プレイヤーの常駐形態（バー型 / フローティング / 別の発明）
- 作品プレビューと詳細ビューの分け方（オーバーレイ / 別画面 / その他）
- ライト/ダークの基本方針

### 委ねないこと（動かさないでほしい）

- **扱うデータ構造**（4 章）。タグの 2 種類、URL が複数、プレイリストが複数あり得る、物理ファイルがトラック以外にも存在する、といった前提
- **ユースケースの優先順**（3 章）。3.1〜3.3 が日々もっとも頻繁に使われる動線
- **シングルユーザー前提**。ログイン / 共有 / コメント / SNS 連携などの概念は不要
- **ローカル動作**。クラウド同期やオンラインサービス的な装いは避ける（カバー画像以外の外部リソースを多用しない）

## 10. 採用されたデザイン方針（2026年5月確定）

本ブリーフに基づき Claude Design で作成したモック (`design_handoff_mimimilli_library/`、以下 "mimimilli") を正典とするデザインシステムが採用された。以下はその決定内容の記録。

**正典デザインファイル:** `design_handoff_mimimilli_library/README.md` および `milli/tokens.css`

**UI構成:** 固定4行グリッドシェル（トップバー / アドレスバー / 本体 / 再生バー）＋左ナビ。本体は3ペイン固定（軸レール → コンテンツ列 → プレビュー）

**テーマ:** ライト基調（紙色トーン・oklchトークン）。ダーク切り替えは将来フェーズ

**プレイヤー:** 単一常駐トランスポート（マルチch ミキサーは将来フェーズ・UIに枠のみ）

**採用した新機能:** 軸ピボット（サークル/CV/タグ別並べ替え）・タグAND積集合絞り込み・スマートフォルダー（保存クエリ）

**延期した新機能:** マルチch ミキサー・再生統計/レーティング/メモ・File Explorer モード（左ナビに枠のみ）

詳細スコープは `docs/requirements-v4.md` のフェーズ定義を参照。
