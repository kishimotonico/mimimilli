# 実UIベースのUI/UX調査メモ

作成: 2026-07-03 / Codex（agent-browser実地確認）

## 前提

- 対象は `http://mimi.localhost:1355` で実際に動いているフロントエンド。
- モックとの差分は見ない。動作中のUIを出発点に、見やすさ・分かりやすさ・既存操作の摩擦を確認した。
- 未実装UIが残っていること自体は今回の問題として扱わない。ビュー切替、左ナビ横断サーフェス、スマートフォルダー条件追加などは将来機能として許容する。
- agent-browser セッションは `mimimilli-ux-review`。スクリーンショットは `tmp/mimimilli-ux-review/screenshots/` に保存した。`tmp/` はGit管理対象外の作業証跡として扱う。

## 全体所感

前回のUI改善後、作品詳細・タグ追加・再生バー/ポップアップの基本操作はかなり整理されている。
大きなレイアウト崩れやJSエラーは確認できなかった。
一方で、機能追加前に潰す価値がある摩擦はまだ残っている。
特に「検索結果が空になったときの復帰しづらさ」「全画面プレイヤーのモーダルとしての不完全さ」「即時編集操作の誤操作耐性」は、実装済み機能の体験品質に直結する。

## 改善候補

### 1. 検索0件時の空状態が復帰しづらい

検索語 `zzzz` で0件にすると、中央カラムに「作品が見つかりません」だけが表示される。
検索語が原因なのか、軸やタグの絞り込みが原因なのか、どの操作で戻せるのかが画面から分かりにくい。
検索欄内のクリアボタンもないため、キーボードに慣れていないユーザーは復帰操作を見つけづらい。

改善案:

- 0件表示に検索語を含める。「`zzzz` に一致する作品はありません」のように原因を明示する。
- 「検索をクリア」ボタンを空状態と検索欄内の両方、またはどちらかに置く。
- タグ/軸フィルタが併用されている場合は、現在効いている条件も表示する。
- 0件時も右ペインに「検索条件を変える」などの同じ文脈の案内を出し、空白感を減らす。

優先度: 高。
既存検索の分かりやすさに直結し、仕様追加なしで改善できる。

### 2. 全画面プレイヤーが視覚上は全画面だが、操作モデルとしては背面が残っている

全画面プレイヤー表示中も、アクセシビリティツリーには背面の検索欄・ナビ・作品詳細・ポップアップの要素が残っていた。
Tab移動でも背面要素へ到達し得る状態に見える。
見た目は全画面なのに、キーボード/スクリーンリーダー上はモーダルになり切っていない。

改善案:

- 全画面プレイヤー表示中は背面のアプリ本体を `inert` 相当にする。
- `aria-modal="true"` と `role="dialog"` もしくは、画面全体を占有するプレイヤーとして適切なランドマークを付ける。
- フォーカス初期位置を「縮小」または主再生ボタンに移す。
- Tab/Shift+Tab をプレイヤー内に閉じ込める。
- Escで閉じた後は、開く前のボタンへフォーカスを戻す。

優先度: 高。
HANDOFFの「フォーカストラップ無し」と同じ領域だが、背面の `inert` 化まで含めて扱うとよい。

### 3. 全画面プレイヤーのアイコンボタンにアクセシブル名がない

全画面プレイヤー内の前/次/ループなどのアイコンボタンが、agent-browserのスナップショットでは `button` のみとして出ていた。
視覚的にはアイコンで理解できるが、スクリーンリーダー・ツールチップ・テスト上では意味が失われる。
ポップアップ側は「前のトラック」「次のトラック」「ループ」などの名前が出ているので、全画面側だけ一貫性が弱い。

改善案:

- 全画面プレイヤーの各トランスポートボタンに `aria-label` と `title` を付ける。
- ループなど状態を持つボタンは `aria-pressed` も付ける。
- `IconButton` に収まらない専用円形ボタンでも、アクセシブル名の付与ルールは共通化する。

優先度: 高。
小さな修正で、分かりやすさとテスト容易性が上がる。

### 4. タグ削除が常時露出しており、即時編集としては誤操作耐性が弱い

作品詳細ではタグチップの削除ボタンが常に表示され、クリックすると即時保存される。
タグ即時編集の方針自体は良いが、分類/整理情報をワンクリックで失う操作としては少し軽い。
特にタグチップ列の近くに「タグを追加」ボタンや再生アクションが並ぶため、細かいクリック操作で誤削除の可能性がある。

改善案:

- 削除ボタンはチップhover/focus時に強調し、通常時は薄くする。
- 削除後に短い undo toast を出す。「タグ `後輩` を削除しました / 元に戻す」。
- 保存中/失敗時の状態をチップ単位で出す。失敗時に黙って元に戻るだけだと分かりにくい。

優先度: 中〜高。
即時編集UIの品質を上げるなら、undoが最も効く。

### 5. タグ追加ポップオーバーが右端に寄り、狭い状況で窮屈に見える

タグ追加ポップオーバーは右ペイン内に収まっているが、右端に近く、入力欄と候補リストの横幅がかなり狭く見える。
短いタグでは問題ないが、構造化タグや長い日本語タグを入力すると読みづらくなる可能性が高い。

改善案:

- ポップオーバーの最小幅を広げる。右端に寄る場合は左方向へ展開する。
- 候補行はタグ名と補足情報の優先順位を明確にし、長い値は中略 + tooltip にする。
- 右ペイン幅が一定以下なら、タグ追加UIをチップ列下のフル幅行として展開する。

優先度: 中。
既に機能は動いているので、レイアウトの磨きとして扱う。

### 6. 900px幅でスマートフォルダーのルール表示が詰まる

900x720 の表示では、スマートフォルダーのルール値が横方向に詰まり、「カテゴリ ASMR OR 環境音」の関係が読み取りづらくなった。
現状はカードとしては崩れていないが、条件を理解するためのUIとしては情報密度が高すぎる。

改善案:

- ルール行を `field / operator / values` の1行固定にしすぎず、値チップは折り返す。
- 狭幅では `WHERE 長さ ≥ 1:00:00` のような自然文寄り表示に切り替える。
- `OR` は小さなラベルではなく、値チップ間の区切りとして視認しやすくする。

優先度: 中。
スマートフォルダー編集前に、まず読み取り表示の堅牢性を上げるとよい。

### 7. 並び替えメニューの選択表示がチェックとして少し読み取りづらい

並び替えメニューは機能している。
ただし選択中マークが小さく、チェックなのか装飾なのかやや判別しづらい。
メニュー自体が小さいので、選択状態はもう少し明快な方がよい。

改善案:

- 専用のチェックアイコンを使う。
- 選択中行の背景や文字色を軽く変える。
- 現在の並び替え名をボタン横またはtooltipに含める。

優先度: 低〜中。
致命的ではないが、繰り返し使う操作なので磨く価値はある。

### 8. トラック行のクリック領域と右端再生ボタンの役割が重複している

作品詳細のトラック一覧は、行全体クリックでも再生でき、右端にも再生ボタンがある。
どちらも同じ操作なら、ユーザーにとっては「行選択」と「再生」が別の意味を持つのか判断しづらい。
小さな再生ボタンは視覚的な affordance として役立つが、常時表示すると密度が上がる。

改善案:

- 行全体クリックを再生に統一し、右端ボタンはhover/focus時だけ表示する。
- 逆に、行クリックは選択/詳細表示、右端ボタンは再生に分けるなら、選択状態と再生状態を明確に分ける。
- 現状のままなら、右端ボタンの見た目を「補助」程度に弱める。

優先度: 中。
トラックが多い作品ほど効く。

### 9. 開発用DevtoolsボタンがプレイヤーUIと競合する

TanStack Query Devtools の右下ボタンが、再生バーのカバーやポップアップ、全画面プレイヤーの右下と重なる。
開発環境限定の問題だが、UI確認・スクリーンショット・クリック検証ではノイズになる。

改善案:

- Devtoolsボタンの初期位置を左下や上部に逃がす。
- プレイヤー表示中だけDevtoolsの位置を変える。
- ビジュアル検証時はDevtoolsを無効化するフラグを用意する。

優先度: 低〜中。
ユーザー向けではないが、今後のUI検証品質に影響する。

## 優先して着手するなら

1. 検索0件の空状態と検索クリア導線。
2. 全画面プレイヤーの `inert` / フォーカストラップ / アクセシブル名。
3. タグ削除のundoまたは誤操作耐性。
4. 狭幅でのスマートフォルダールール表示。
5. トラック行と再生ボタンの役割整理。

この順なら、未実装機能を進めなくても、既存の主要導線の分かりやすさを底上げできる。
