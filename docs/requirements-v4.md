# ローカル音声作品管理・再生アプリ 要件定義

## 1. 概要

### 1.1 背景

DLsiteやDMMのWebプレイヤーが使いにくく、横断的な検索ができない。ローカルに保存した音声作品を快適に管理・再生するアプリが必要。

### 1.2 コンセプト

**タグによる分かりやすい検索性**と、**物理的なフォルダーによる手動での管理性**の両立。

### 1.3 ターゲットプラットフォーム

- v1: Windows / macOS / Linux でローカル起動する Web アプリ（ブラウザ + ローカル API サーバー）
- 将来: リモート端末（iPad等）からの LAN ストリーミングを視野に入れる

### 1.4 技術スタック

- **フロントエンド:** React 19 + TypeScript + Vite
- **バックエンド:** Rust + axum（ローカル HTTP API サーバー）
- **音声再生:** HTML5 Audio + Web Audio API（フロントエンド完結）
  - 将来的に高度な機能（ギャップレス再生等）が必要になった場合、Rust側への移行余地を残す
- **データベース:** SQLite (rusqlite bundled)
- **設計方針:** Rust 側にサービス層（APIレイヤー）を設け、HTTP ハンドラはそのラッパーとする。ローカルUIとリモートUIで同じサービス層を共有できる設計。

### 1.5 対応音声フォーマット

HTML5 Audio（ブラウザの Chromium エンジン）のサポート範囲に準拠する。

**フェーズ1 対応フォーマット:** MP3, AAC/M4A, WAV, OGG (Vorbis), FLAC, WebM (Opus)

**非対応（将来検討）:** WMA, ALAC, DSD 等

DLsite作品の大半はMP3かWAV、FANZAはM4Aが多いため、上記で実用上ほぼカバーできる。非対応フォーマットのファイルはトラックリスト上でグレーアウト＋非対応アイコン表示とし、再生しようとした場合はエラーメッセージを表示する。

## 2. 作品の管理

### 2.1 作品の形式

作品は以下の2形式をサポートする。

**フォルダー形式:** 販売サイトからダウンロードしたファイル一式が1フォルダーにまとまった形式。フォルダー内には複数の音声ファイル、画像ファイル、PDF、テキストファイルが含まれる。1つの作品内にSE有無やフォーマット違い（mp3/wav）など複数のバリアントが存在するケースがある。

**単一ファイル形式:** 音声作品の全トラックを結合した1ファイル（mp3, m4a等）。チャプター情報に基づきトラックごとに再生する。

### 2.2 フォルダー構造

ユーザーが指定したルートフォルダー以下を管理対象とする。ルート直下に全作品がある必要はなく、ユーザーが自由なフォルダー構造で保存できる。

```
root/
├── dlsite/
│   ├── genre-a/
│   │   └── RJ1234/              ← フォルダー形式の作品
│   │       ├── SE有り/mp3/
│   │       ├── SE無し/mp3/
│   │       ├── cover.jpg
│   │       └── .meta.json
│   └── genre-b/
│       └── RJ5678/
└── fanza/
    ├── d00001.mp3                ← 単一ファイル形式の作品
    └── d00001.meta.json
```

### 2.3 作品の認識とスキャン

作品フォルダー内、または単一ファイルと同階層に**メタファイル（`.meta.json`）**を配置することで作品として認識する。スキャン時にメタファイルを検出し、内部DBと同期する。

### 2.4 作品IDと移動追従

メタファイル内にUUID形式の作品IDを保持する。ユーザーがフォルダーやファイルを物理的に移動・リネームした場合、スキャン時にIDで突合しDBのパス情報を更新する。DB上に存在するがスキャンで見つからない作品は「行方不明」ステータスとする。

> **フェーズ1では移動追従のIDベース突合は実装しない。** フォルダーを移動した場合は再スキャンで対応する運用とする。

## 3. メタデータ

### 3.1 二重管理方式

メタファイル（`.meta.json`）をSource of Truthとし、内部DB（SQLite）は検索・一覧のパフォーマンス用キャッシュとして運用する。DB側で編集した場合はメタファイルに即時書き戻す（DB編集とメタファイル更新を同一操作内で行う）。

アプリが起動していない間にユーザーがメタファイルを手動編集した場合、次回スキャン時にメタファイルの内容でDBを上書きする（メタファイルが常に勝つ）。

複数のメタファイルに同一UUIDが存在した場合、後に検出された方を新規UUIDで再採番し、ユーザーに通知する。

### 3.2 メタファイル フォーマット

```jsonc
{
  "id": "uuid-xxxx-xxxx-xxxx",
  "title": "作品タイトル",
  "urls": [
    { "label": "DLsite", "url": "https://www.dlsite.com/..." },
    { "label": "FANZA", "url": "https://www.dmm.co.jp/..." }
  ],
  "tags": [
    "R-15",
    "バイノーラル",
    "cv/ひなたゆか",
    "サークル/ほげふが"
  ],
  "coverImage": "cover.jpg",
  "playlists": [
    {
      "name": "default",
      "tracks": [
        { "title": "導入", "file": "SE無し/mp3/01_導入.mp3" },
        { "title": "本編", "file": "SE無し/mp3/02_本編.mp3" }
      ]
    }
  ],
  "defaultPlaylist": "default",
  "createdAt": "2025-01-01T00:00:00Z"
}
```

`urls` は作品に関連するURLの配列。販売ページ（DLsite, FANZA）、作者のSNS、公式サイトなどを複数登録できる。通常は1〜2件、多い場合で3〜5件を想定。`label` はUI表示用の短いラベル。空配列の場合はURLなしとして扱う。

### 3.3 トラック定義

トラックのスキーマはフォルダー形式・単一ファイル形式で統一する。各トラックは「どのファイルのどの区間を再生するか」を表す。

```jsonc
// フォルダー形式: ファイルパスのみ（区間省略 = ファイル全体）
{ "title": "導入", "file": "SE無し/mp3/01_導入.mp3" }

// 単一ファイル形式: 同一ファイルに対して区間指定（秒）
{ "title": "導入", "file": "d00001.m4a", "start": 0, "end": 180 }
```

`start`/`end` が省略されている場合はファイル全体を再生する。これによりプレイヤー側のロジックは常に「指定ファイルの指定区間を再生する」の1つで統一される。単一ファイルのチャプター情報のソースは、ファイル内蔵のチャプターマーカーまたはユーザー手動設定とする。`tracks` が未定義の場合は1トラック（ファイル全体）として扱う。

> **フェーズ1ではチャプター自動読み取り（M4A/MP3のチャプターマーカー解析）は実装しない。** 単一ファイル形式はファイル全体を1トラックとして扱い、手動でのチャプター設定UIもフェーズ2以降とする。

### 3.4 作品内プレイリスト

`playlists` 配列により、1作品に対して複数のトラック構成を持てる。SE有無やフォーマット違いなどのバリアントに対応する。

```jsonc
"playlists": [
  {
    "name": "SE無し",
    "tracks": [
      { "title": "導入", "file": "SE無し/mp3/01_導入.mp3" },
      { "title": "本編", "file": "SE無し/mp3/02_本編.mp3" }
    ]
  },
  {
    "name": "SE有り",
    "tracks": [
      { "title": "導入", "file": "SE有り/mp3/01_導入.mp3" },
      { "title": "本編", "file": "SE有り/mp3/02_本編.mp3" }
    ]
  }
],
"defaultPlaylist": "SE無し"
```

大半の作品ではプレイリストは1つのみの想定。複数プレイリストは一部の作品で必要になるため、データ構造としては対応するが、**UIでの複数プレイリスト管理機能はフェーズ2以降で実装する**。フェーズ1では `defaultPlaylist` の1つのみ再生可能とする。

### 3.5 メタファイルの自動生成

スキャン時にメタファイルが存在しないが音声ファイルを含むフォルダーを検出した場合、メタファイルの自動生成を行う。自動生成はあくまで下書きであり、ユーザーが後から修正する想定。

**自動生成の内容:**

- `id`: UUID自動採番
- `title`: フォルダー名から取得（RJコードがフォルダー名に含まれる場合はDLsiteの作品コードとして認識・保持する）
- `urls`: 空配列（手動追加に委ねる。フェーズ2ではRJコードからDLsiteのURLを自動設定）
- `tags`: 空配列（手動追加に委ねる）
- `coverImage`: フォルダー内の画像ファイルを探索し、あれば最初の1枚を設定（なければnull）
- `playlists`: フォルダー内の音声ファイルをファイル名の自然順ソートで列挙し、`default` プレイリストとして生成。サブフォルダーが複数ある場合（`SE有り/`, `SE無し/` 等）は、最もファイル数が多いサブフォルダーの音声ファイルを `default` プレイリストとする

単一ファイル形式（ルートフォルダー直下等にある単独の音声ファイル）については、自動生成の対象外とする。ユーザーがUI上から手動でメタファイルを作成する。

### 3.6 DB固有の情報（メタファイルには保存しない）

以下の情報はDBのみで管理する。

- 作品の物理パス（スキャン時に更新）
- 再生位置（レジューム情報）— 将来実装
- 作品のステータス（正常 / 行方不明 / エラー）
- デフォルトプレイリストの合計再生時間（スキャン時またはメタデータ更新時に算出しキャッシュ）
- ブックマーク状態（将来実装）
- 最終再生日時（将来実装）

## 4. タグシステム

### 4.1 タグの種類

**フラットタグ:** 単純な文字列。ジャンルや属性を表す。例: `R-15`, `バイノーラル`, `触手`, `耳かき`

**Annotatedタグ:** スラッシュ区切りで構造的に管理するタグ。`カテゴリ/値` の形式。最初のスラッシュでのみ分割する（値にスラッシュが含まれる場合でも正しく処理される）。例: `cv/ひなたゆか`, `サークル/ほげふが`, `シリーズ/○○物語`

### 4.2 タグの運用

作品のほぼすべての属性（サークル名、CV、ジャンル等）をタグで管理する。1つの作品に対して多数のタグが付く想定。ユーザーが自由にタグを定義・追加・削除できる。

### 4.3 DB設計とパフォーマンス

タグ検索はSQLiteの多対多リレーション（works, tags, work_tags）で実装する。数千作品 × 作品あたり数十タグ（中間テーブル数万行）の規模であれば、適切なインデックスにより十分なパフォーマンスを確保できる。annotatedタグの前方一致検索（`cv/%` でCV一覧等）も対応可能。

### 4.4 DLsiteからのタグ自動取得（フェーズ2）

DLsiteの作品ページからスクレイピングしてタグを自動取得する。DLsiteのジャンルタグはprefix付きで変換する（例: `genre/耳かき`, `genre/バイノーラル`）。取得対象はタイトル、サークル名、CV、ジャンルタグ、カバー画像など。RJコードがフォルダー名に含まれる場合、DLsiteの作品コードとして扱う。

## 5. プレイヤー機能

### 5.1 フェーズ1 実装機能

- 基本再生: 再生 / 一時停止 / 停止
- シーク（再生位置の移動）
- 音量調整
- トラック切り替え（前へ / 次へ）
- 作品内の連続再生
- ±10秒ジャンプ（ASMR作品では再生位置の微調整が重要なため、標準コントロールとして提供）
- ループ再生（現在のトラックをリピート）

### 5.2 プレイヤーの表示モード

プレイヤーは以下の3つの表示モードを持つ。

**プレイヤーバー（固定）:** 画面下部に固定表示されるコンパクトなコントロール。再生中のトラック情報、シークバー、基本操作を備える。再生中は全画面で常駐する。フェーズ1ではこの固定バーのみ実装する。

**プレイヤーバー（フローティング）:** 画面右下にフローティング表示。ドラッグで位置変更可能。DLsite Playのプレイヤーに近い形式。フェーズ2で実装し、固定バーとの切り替えを可能にする。

**フル画面プレイヤー:** プレイヤーバーをクリックすると展開。カバー画像を大きく表示し、トラックリスト全体の操作が可能。閉じるとバーに戻る。

### 5.3 再生モード

**トラックモード:** メタファイルのプレイリスト（`defaultPlaylist`）に基づき、音声ファイルをトラックリストとして表示。トラック単位の再生が可能。

> **フォルダービューはフェーズ2以降で実装する。** フェーズ1ではトラックモードのみ。

### 5.4 将来実装機能

- L/R入れ替え（Web Audio APIの ChannelSplitter/Merger で実装）
- フォルダービュー（作品フォルダーの物理構造を表示し、音声・画像・PDFを閲覧）
- A-Bリピート（区間繰り返し再生）とポイントの保存
- レジューム（再生位置記憶、DB保存）
- バックグラウンド再生 + グローバルホットキー
- ギャップレス再生（Rust側での実装を検討）
- 再生速度変更
- プレイリスト（作品横断）

## 6. エラーハンドリング

### 6.1 方針

エラーが発生した作品はライブラリ一覧のカバー画像上にエラーアイコンを重ねて表示し、ユーザーに修正を促す。作品詳細画面でエラーの具体的な内容を表示する。

### 6.2 検出するエラー

- **メタファイル不正:** JSONパースエラー、必須フィールドの欠落
- **音声ファイル欠損:** メタファイルで定義されたトラックの参照先ファイルが存在しない
- **音声ファイル破損:** 再生時にデコードエラーが発生した場合
- **非対応フォーマット:** サポート外の音声フォーマットのファイルがトラックに含まれる
- **カバー画像欠損:** `coverImage` で指定されたファイルが存在しない（この場合はデフォルトのプレースホルダー画像を表示）

## 7. UI / 画面構成

> UIの詳細設計は別途UI設計書として管理する。以下は画面構成の概要。

### 7.1 画面一覧

**初期セットアップ画面:** 初回起動時のみ。ルートフォルダー選択→初回スキャン→ライブラリ一覧へ遷移。

**ライブラリ一覧画面（メイン画面）:** 作品のグリッド表示またはテーブル表示。検索バー（テキスト検索＋タグフィルター統合）、検索条件バー（現在の条件表示＋並び替え）、グリッドサイズ調整、表示モード切り替えを備える。

**クイックビュー（右パネル）:** ライブラリ一覧でのシングルクリックで表示されるオーバーレイパネル。カバー画像、メタ情報、タグ、トラックリスト、URLリンクを表示。素早い再生開始やタグ編集に使う。

**作品フルビュー:** ダブルクリックまたはクイックビューの「詳細を開く」で遷移。画面全体を使い、左サイドバーにカバー画像＋メタ情報、右側にトラックタブとファイルタブを持つ。ファイルタブではフォルダーツリー形式で物理ファイル構造を表示し、音声の再生や画像のプレビューが可能。

**プレイヤーバー:** 画面下部に固定。再生中のみ表示。クリックでフル画面プレイヤーに展開。

**フル画面プレイヤー:** カバー画像大表示、再生コントロール、トラックリスト全体。

**設定モーダル:** ルートフォルダーの変更、フルスキャンの実行。

**新規作品検出ポップアップ:** 新しい作品が検出された際にタイトル確認・登録を行うダイアログ。

### 7.2 ライブラリ一覧の表示

**グリッド表示:** カバー画像のグリッド。カバーサイズを4段階（S/M/L/XL）で可変。各カードにタイトル、トラック数、合計再生時間を表示。エラーのある作品はカバー画像上にエラーアイコン、ブックマーク済み作品はブックマークアイコンを重ねる。

**テーブル表示:** タイトル、タグ、トラック数、合計再生時間を一覧形式で表示。ミニカバー画像付き。フェーズ2で実装。

### 7.3 検索・フィルタリング・並び替え

**検索条件バー:** ライブラリ一覧の上部に表示。現在の検索ワードとタグフィルターをチップ形式で表示し、個別に削除または一括クリアが可能。結果件数と並び替えコントロールもここに配置する。

**フェーズ1の検索:** テキスト検索（タイトル・タグの部分一致）。

**フェーズ2の検索:** AND/OR/除外、検索条件のプリセット保存。

**並び替え:** 追加日（新しい/古い順）、タイトル順（昇順/降順）、再生時間順（長い/短い順）。フェーズ2で最近再生した順、ランダム順、ID順を追加。

### 7.4 プレイヤーUI

DLSite Play（DLsiteの公式プレイヤー）の操作感をベースとする。ASMR作品の特性を考慮し、シークバーは十分な幅を確保する。再生コントロール、シークバー、音量、±10秒ジャンプ、ループ再生トグルを備える。

## 8. スキャンと初期セットアップ

### 8.1 初期セットアップフロー

1. 初回起動時、ルートフォルダーが未設定であれば設定画面に誘導
2. ユーザーがルートフォルダーを指定
3. ルートフォルダー以下を再帰スキャン（自動実行）
4. メタファイルが見つかった作品はDBに登録/更新
5. メタファイルがない作品候補（音声ファイルを含むフォルダー）はメタファイルを自動生成（3.5項参照）
6. スキャン結果のサマリーを表示（登録済み: N件、新規生成: N件、エラー: N件）

### 8.2 スキャンのトリガーと実行モデル

**フェーズ1のスキャントリガー:**

- **手動トリガー:** 設定モーダル内に「フルスキャン実行」ボタンを配置。前回スキャン日時も表示する
- **ヘッダーの再読み込みボタン:** ライブラリ一覧のヘッダーに簡易的な再読み込みボタンを配置。軽量な再スキャンまたはDB再読み込みに使用する
- **初回起動時:** ルートフォルダー設定完了後に自動でフルスキャンを実行

**新規作品の検出:** スキャンで新しい作品候補が見つかった場合、確認・登録用のポップアップを表示する。ポップアップではタイトルの確認・編集が可能。フェーズ2ではスクレイピング結果（タグ、カバー画像等）もここに表示する。

ファイル監視（watcherによるリアルタイム検知）やアプリ起動時の自動スキャン、差分スキャンはフェーズ2以降で検討する。

**スキャン中のUI:** スキャンはバックグラウンドで実行し、スキャン中もライブラリの閲覧・再生は可能とする。完了したら結果のサマリーをトースト通知で表示する。

### 8.3 再スキャン

作品IDによるパス突合で移動・リネームに追従する機能はフェーズ2以降で実装する。フェーズ1ではフルスキャンのみ対応し、フォルダーを移動した場合は再スキャンで新規作品として登録される（旧パスの作品は「行方不明」ステータスとなる）。

## 9. リモートストリーミング

### 9.1 概要

PC上のローカル API サーバー（axum）を常駐させ、iPadなど別端末のWebブラウザからLAN経由で音声をストリーミング再生する。基本的なローカルサーバー化（`apps/server`）は実装済み。

### 9.2 アーキテクチャ

```
[PC (axum サーバー)]                   [iPad等 (Webブラウザ)]
┌──────────────────────┐               ┌──────────────────┐
│  Rust バックエンド     │◄── HTTP ──────│  Web UI (React)  │
│  ┌─────────────────┐ │               └──────────────────┘
```
